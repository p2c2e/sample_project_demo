# Observability

This document covers observability concepts, what this repo demonstrates, and links to official documentation.

## The Three Pillars

| Pillar | What it tells you | This repo |
|--------|------------------|-----------|
| **Traces** | How a request flows through services, where time is spent | Yes -- full distributed tracing |
| **Metrics** | Aggregate measurements (request rate, error rate, latency) | Not yet -- see Next Steps |
| **Logs** | Discrete events with context | Not yet -- see Next Steps |

This demo focuses on **traces** because they are the most visually compelling way to understand distributed systems.

## OpenTelemetry Concepts

Each concept below is defined and mapped to where it appears in this repo's code.

### Trace

A trace represents a single end-to-end request. It is a tree of spans sharing the same trace ID. In this repo, a `GET /api/order/1` request creates a trace spanning all three services.

### Span

A span represents a unit of work within a trace. Each span has a name, start/end time, and optional attributes. Examples in this repo:
- `GET /order/1` (auto-created by FlaskInstrumentor)
- `process-item` (manually created in api-gateway/main.py)
- `enrich-user` (manually created in backend-service/backend_service.py)
- `build-order` (manually created in api-gateway/main.py)

### SpanContext

The immutable data carried across service boundaries: trace ID, span ID, and trace flags. Encoded in the `traceparent` HTTP header.

### TraceId

A 128-bit (32 hex char) identifier shared by all spans in a trace. Generated by the first service that starts the trace.

### SpanId

A 64-bit (16 hex char) identifier unique to each span. Generated by the service that creates the span.

### TracerProvider

The factory that creates tracers. Configured with a Resource (service name) and SpanProcessors.
- Python: `TracerProvider` in `api-gateway/otel_setup.py` line 18
- Node: `NodeTracerProvider` in `tracing.js` line 29

### SpanProcessor

Handles completed spans. `BatchSpanProcessor` batches spans and sends them periodically (more efficient than sending one at a time).
- Python: `BatchSpanProcessor` in `api-gateway/otel_setup.py` line 22
- Node: `BatchSpanProcessor` in `tracing.js` line 32

### Exporter

Sends spans to a backend. `OTLPSpanExporter` sends spans via the OTLP HTTP protocol.
- Python: `OTLPSpanExporter` in `api-gateway/otel_setup.py` line 22
- Node: `OTLPTraceExporter` in `tracing.js` line 24

### Instrumentation

Libraries that auto-create spans for frameworks and HTTP clients.
- Python: `FlaskInstrumentor` + `RequestsInstrumentor` in `api-gateway/otel_setup.py` lines 27-28
- Node: `HttpInstrumentation` + `ExpressInstrumentation` in `tracing.js` lines 42-43

### Propagator

Injects/extracts trace context across service boundaries. The default is the W3C TraceContext propagator, which uses the `traceparent` header. This is configured automatically -- no explicit code needed.

## W3C Trace Context

The `traceparent` header carries trace context across HTTP boundaries:

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
```

| Field | Length | Value | Meaning |
|-------|--------|-------|---------|
| version | 2 char | `00` | Always 00 for current spec |
| trace-id | 32 char | `4bf92f...4736` | Shared across all spans in the trace |
| parent-span-id | 16 char | `00f067...02b7` | The span that made this call |
| trace-flags | 2 char | `01` | 01 = sampled (record this trace) |

In this demo, `RequestsInstrumentor` (Python) and `HttpInstrumentation` (Node) automatically inject this header on outgoing requests. `FlaskInstrumentor` and `ExpressInstrumentation` automatically extract it from incoming requests.

## Auto-Instrumentation vs Manual Spans

### Auto-instrumentation

Automatic spans created by instrumentor libraries. You get these for free:
- Every incoming Flask/Express HTTP request
- Every outgoing HTTP call via `requests` (Python) or `http.request` (Node)

These give you the skeleton of the trace -- you can see which services talked to each other and how long each call took.

### Manual spans

Custom spans you create in your code for business-level visibility:

```python
with tracer.start_as_current_span("process-item") as span:
    span.set_attribute("item.id", item_id)
    # your business logic here
```

Use manual spans when you need to see:
- Which part of a handler is slow (DB query vs computation vs external call)
- Business context (item ID, user ID, order amount)
- Sub-operations within a single HTTP handler

## What to Observe in Jaeger

Open http://localhost:16686, select the **ui-app** service, and click **Find Traces**.

### Single-service trace: GET /items

```
ui-app: GET /api/items
  ui-app: HTTP GET localhost:5001/items
    api-gateway: GET /items
```

Three spans: incoming Express request, outgoing HTTP call, incoming Flask request.

### Three-service trace: GET /api/order/1

```
ui-app: GET /api/order/1
  ui-app: HTTP GET localhost:5001/order/1
    api-gateway: GET /order/1
      api-gateway: HTTP GET localhost:5002/users/1
        backend-service: GET /users/1
          backend-service: enrich-user
      api-gateway: build-order
```

Seven spans across three services. The Jaeger waterfall view shows:
- Total request duration from browser to final response
- Time spent in each service
- Parallel vs sequential operations
- Custom span attributes (click any span to see details)

### Things to look for

- **Span nesting**: child spans are indented under their parents
- **Service colors**: each service gets a different color in the waterfall
- **Duration bars**: wider bars = more time spent
- **Span attributes**: click a span to see `item.id`, `user.name`, etc.
- **HTTP details**: status codes, URLs, methods on auto-generated spans

## Next Steps

Ideas for extending observability in this repo:

1. **Add metrics** -- Use `MeterProvider` to track request count, error rate, and latency histograms. Export to Prometheus or Jaeger (which supports OTLP metrics).

2. **Add structured logging** -- Use Python `logging` with trace correlation. OTel can inject trace ID and span ID into log records so you can jump from a log line to its trace in Jaeger.

3. **Export to Grafana/Tempo** -- Replace Jaeger with Grafana Tempo for traces + Grafana for dashboards. The OTLP exporter works with any OTLP-compatible backend.

4. **Add span events** -- Record discrete events within a span (e.g., cache miss, validation failure). Events appear as annotations in the Jaeger waterfall.

5. **Add span exceptions** -- Use `span.record_exception(error)` to attach error details to spans. Jaeger highlights errored spans in red.

6. **Sampling** -- Configure `TraceIdRatioBased` or `ParentBased` samplers to reduce trace volume in production while keeping interesting traces.

7. **Baggage** -- Propagate key-value pairs across services using OTel Baggage (separate from trace context). Useful for passing tenant ID, feature flags, etc.
